# Design — v0.2 Documentation Detection Engine

> Status: **APPROVED (architecture) — amended, pending final review of this revision.**
> No implementation until the amended design is confirmed.
> Audience: maintainers. Written in English (the project's "language of thought"; see
> PROJECT_PRINCIPLES §3).
>
> **Locked decisions (approved):**
> 1. The interruption policy lives in a dedicated `references/interruption-policy.md`, kept out
>    of SKILL.md to preserve the thin-orchestrator pattern.
> 2. Evaluation fixtures live in a top-level `tests/` directory — behavioral evaluation is a
>    first-class concern from day one.
>
> **Amendment 1:** the engine produces a structured **Evidence object** as the shared substrate
> for scoring, classification, confidence, the user-facing explanation, and future read-side
> features (Resume/Interview). See §4.1.
>
> **Amendment 2 (this revision):** separate **facts** from **judgments**. *Evidence* holds only
> **purpose-independent observations**; a derived ***Assessment*** holds the purpose-relative
> judgments (importance, confidence, classification, explanation). The same Evidence can yield
> different Assessments for different purposes (documentation, resume, interview, portfolio).
> The dividing line is **purpose-independence**, not "zero interpretation." See §4.1–§4.2.

The Detection Engine answers exactly one question:

> **"Should this development event be documented?"**

…and, when the answer is yes, hands off *what kind* and *how valuable* to the existing v0.1
capture workflow. Nothing more.

---

## 0. Framing: this is not a new layer

The single most important design decision, so it comes first.

**Decision:** The Detection Engine is realized as an *enhanced set of the existing
`references/` documents* that the `second-brain` skill already consults — not a new subsystem,
service, or architectural layer.

- **Why:** PROJECT_PRINCIPLES §4/§8 forbid new layers unless absolutely necessary, and §6 says
  behavior should evolve by *refining references, not swelling the orchestrator*. v0.1 already
  has the two seeds of a detection engine: `detection-rules.md` (record / don't) and
  `importance-score.md` (⭐1–5). The engine is what these become when made rigorous.
- **Alternatives considered:**
  - *A standalone detection component/script.* Rejected: there is no runtime code in v0.1
    (behavior is prompt-driven); a code component would introduce a layer the constitution
    warns against and couple detection to an execution model we don't have yet.
  - *A second skill dedicated to detection.* Rejected: detection is not a separate user-facing
    capability — it is the front half of the capture skill's own workflow. Splitting it would
    fragment one concern across two skills (violates §8 "adding a feature shouldn't require
    understanding all others").
- **Why preferable:** Keeps the thin-orchestrator shape intact, reuses the loaded-on-demand
  reference mechanism, and means v0.2 ships as *better prompts + test data*, with zero
  structural churn.

**Corollary — decouple the decision from the trigger.** The engine is a pure *decision
function*: given the current context, return a verdict. *When* it runs (skill auto-activation
today; hooks in v0.4) is a separate concern. Designing it as a pure function is what lets v0.4
hooks call the identical engine later without rework.

---

## 1. Goals

1. **Reduce false interruptions.** The primary UX metric. A wrong "record this?" costs the
   user's trust; §3 calls their attention sacred. High precision is the point.
2. **Catch the genuinely valuable events** — hard bugs, real decisions, new tech, architecture
   — reliably enough that the vault reflects real work.
3. **Time proposals well.** Propose at natural stopping points, never mid-task.
4. **Produce an interpretable verdict** — a structured **Evidence object** (which signals fired
   and why), with score and confidence *derived* from it, so a human can sanity-check the
   reasoning. No black boxes (§6: instructions must be testable).
5. **Be reusable by future triggers** (v0.4 hooks) with no change to the engine itself.

*Why these and not "maximize documents captured":* §3 Quality over quantity. The engine's job
is to say **no** well, more than to say yes often.

## 2. Non-goals

- **Not automatic saving.** The engine only ever *proposes*. §3 never-auto-save is absolute.
- **Not a trigger mechanism.** It does not decide *when it runs*; hooks (v0.4) own that.
- **Not writing or classification internals.** It hands off to the existing doc-type selection
  and template-fill steps; it does not reimplement them.
- **Not cross-session memory or analytics.** v0.2 reasons about the current session only.
  Evidence and the documentation Assessment *are* persisted — but only as frontmatter riding the
  saved document, not as a separate decision log or analytics store, and never read back to
  influence a later session's detection. Remembering prior sessions and reports stay later
  concerns (read-side, §8).
- **Not multiple Assessments yet.** The Evidence/Assessment split is introduced now so the
  substrate is right, but v0.2 produces only the `documentation` Assessment. Resume/interview/
  portfolio Assessments are v0.5+.
- **Not deterministic scoring.** Judgment is model-made and will vary; we constrain and
  evaluate it, we do not pretend it is exact.
- **Not multi-language.** Korean output only, per v0.1 limitations.

## 3. Detection pipeline

A cheap-to-expensive funnel. Each stage can terminate early with "stay silent."

```
Event context (recent conversation)
   │
   ▼
① Eligibility gate      → hard-exclude trivial/non-dev work        → [reject → silent]
   │
   ▼
② Evidence extraction   → build the EVIDENCE object (facts, §4.1)  ◀── purpose-independent
   │                         signals fired + support + resolution + explicit_request + negatives
   │
   ├──────────────┬───────────────┬────────────────────────┐   ─┐
   ▼              ▼               ▼                        ▼    │  all of ③④⑥ + explanation
③ Importance    ④ Confidence    ⑥ Classification         (user  │  together form the
   scoring         estimate        (type + title)           expl.)│  ASSESSMENT object
   (⭐1–5, §5)      (§6)            (via doc-types.md)  (derived) ─┘  (judgment, purpose=documentation, §4.2)
   │              │               │
   └──────────────┴───────────────┘
   │
   ▼
⑤ Interruption decision → score × confidence × timing → propose? (§7/§8) → [hold/silent]
   │
   ▼
   Handoff (§9): EVIDENCE + the documentation ASSESSMENT persist in the saved doc's frontmatter
```

Two named artifacts, mapping onto stages the pipeline already had:
- Stage ② produces the **Evidence object** — immutable, purpose-independent facts. "Why, observed."
- Stages ③④⑥ + the explanation together produce an **Assessment object** — the purpose-relative
  judgment. "Why, evaluated (for documentation)."

The same Evidence can later produce *other* Assessments (resume, interview, portfolio) without
re-observing anything. "Why" as *fact* is computed once; "why" as *evaluation* is computed per
purpose.

- **Why a staged funnel:** cheapest, highest-precision rejections happen first (the gate throws
  out typos before any scoring effort), matching §6 "orchestrate, then delegate." It also gives
  clean, testable exit points for evaluation (§11).
- **Alternatives considered:** *A single holistic "is this worth it?" prompt.* Rejected: it
  conflates rejection, scoring, confidence, and timing into one judgment that can't be tuned or
  evaluated per-stage, and tends to over-fire because it has no explicit cheap-reject gate.
- **Why preferable:** separable stages = separately testable and separately improvable, without
  growing any single prompt into a monolith.

## 4. Detection signals & the Evidence object

Signals are *evidence*, gathered in stage ②. They are grouped by what they indicate. None is
individually sufficient (except the explicit request); the scorer weighs them together.
Stage ② assembles them into a structured **Evidence object** (defined at the end of this
section) that the rest of the pipeline consumes.

**A. Explicit intent (decisive).**
- User says "기록해줘 / 정리해줘 / 문서로 남겨줘" or runs `/document`.
- *Why decisive:* the user has already decided. The engine should never argue with a direct
  request; it skips straight to handoff (still requiring approval of the final write).

**B. Difficulty signals** (indicate troubleshooting value).
- Multiple failed attempts before resolution; a long back-and-forth on one problem; an error
  that was non-obvious; "finally / 드디어 / 이게 문제였네" style relief markers.

**C. Decision signals** (indicate ADR value).
- Comparison of alternatives ("X vs Y"), stated trade-offs, an explicit choice with reasons.

**D. Novelty signals** (indicate TIL / study-note / ADR value).
- A new library/framework/tool introduced or adopted; a first-time integration.

**E. Domain signals** (raise prior probability).
- Work touching authentication, DB schema, API design, performance, architecture, caching,
  security — the domains §Detection-rules already names as high-value.

**F. Resolution signals** (indicate a *complete*, documentable event).
- Tests passing after failing; "됐다 / 해결", a fix confirmed working, a milestone reached.

**G. Negative signals** (push toward silence).
- Cosmetic/typo/syntax markers; mid-exploration with no resolution; the user actively still
  debugging; topic already covered in the vault.

- **Why signals rather than keyword rules:** §6 wants interpretable-but-not-brittle logic.
  Signals are *categories of evidence* the model recognizes semantically, not literal string
  matches. A keyword list ("error", "fix") would both over-fire (every message mentions errors)
  and miss paraphrases.
- **Alternatives considered:** *(a) pure keyword/regex triggers* — brittle, high false-positive;
  *(b) fully implicit "you decide"* — unauditable, drifts. **Chosen: named signal categories
  with examples** — the model judges semantically but against an explicit, testable taxonomy.
- **Why preferable:** new signals are added as *data* (another bullet), never as new code paths
  (§8 extensibility).

### 4.1 The Evidence object (shared substrate)

Stage ② emits one structured artifact — the Evidence object — that records **observed,
purpose-independent facts**: which signals fired, their supporting observations, the resolution
state, whether the user explicitly asked, and any negative signals. It contains **no
evaluation** — no importance, no confidence, no type, no explanation. Those are judgments and
live in the Assessment (§4.2). The test for "does this belong in Evidence?" is
**purpose-independence**: would this observation be identical whether we are documenting,
building a resume, or prepping for an interview? If yes, it is a fact → Evidence. If it depends
on the purpose, it is a judgment → Assessment.

**One vocabulary.** The Evidence object uses the *same* signal taxonomy as §4 above — there is
no second list. The example signals raised in review map onto it as follows, so §5's
one-concept-one-home is preserved:

| Review example | §4 signal group |
|----------------|-----------------|
| Difficulty | B. Difficulty |
| Technical Decision, Trade-off | C. Decision |
| Novelty | D. Novelty |
| Learning Value | D. Novelty (TIL/study-note facet) |
| Performance, Architecture | E. Domain |

**Shape (conceptual — a documented structure, not a code type):**

```
Evidence {
  contributing_signals: [
    { group: "difficulty",  support: "3 failed attempts before the 401 loop was traced" },
    { group: "decision",    support: "chose Redis over in-memory for refresh-token storage" },
    { group: "domain",      support: "authentication / session security" }
  ],
  resolution: "confirmed" | "partial" | "none",   // from §4F — is the event finished?
  explicit_request: true | false,                  // §4A
  negatives: [ "…any §4G push-to-silence notes…" ]
}
```

**Guardrails (non-negotiable for this artifact):**

1. **Qualitative, not numeric.** Each contributing signal carries a short natural-language
   `support`, never a numeric weight or score. Per-signal numbers would reintroduce the exact
   false-precision trap rejected for scoring (§5/§6). Strength is conveyed by *what the support
   says*, and by how many independent signals corroborate.
2. **Same taxonomy as §4.** No parallel signal list ever. New signals are added to §4 and become
   available to Evidence automatically.
3. **Lean.** Only signals that actually fired are listed. Evidence is a compact record of the
   decision, not an essay.

- **Why the Evidence object exists:** it makes the observed "why" a *single, immutable,
  purpose-independent artifact* (§5 one home), turns observation into an auditable, testable
  structure (§6), and gives every future purpose a stable contract to read instead of
  re-observing (§8).
- **Why facts-only:** keeping evaluation *out* of Evidence is what lets the same Evidence serve
  documentation today and resume/interview/portfolio later. If Evidence carried a
  documentation-flavored score, every other purpose would have to ignore or overwrite it — the
  contract would leak one purpose into a shared substrate.

### 4.2 The Assessment object (derived judgment)

An Assessment is what Evidence *means for a specific purpose*. For v0.2 the only purpose is
`documentation`; the same Evidence will later yield separate Assessments for `resume`,
`interview`, and `portfolio`.

**Shape (conceptual):**

```
Assessment {
  purpose: "documentation",                       // later: "resume" | "interview" | "portfolio"
  should_document: true,                           // the go/no-go — itself a judgment
  importance: "⭐⭐⭐⭐",                             // §5, from Evidence
  confidence: "high",                              // §6, from Evidence.resolution + signal strength
  classification: { type: "troubleshooting", alternates: ["adr"] },  // §9, via doc-types.md
  explanation: "…rendered from Evidence.contributing_signals…"       // not written independently
}
```

- Every field is **derived from Evidence**; an Assessment never introduces a fact that isn't in
  Evidence. The `explanation` is *rendered from* `Evidence.contributing_signals`, so it cannot
  disagree with the judgment or the facts.
- Different purposes weigh the *same* facts differently: a measurable performance win might be
  ⭐⭐⭐ for `documentation` but decisive for `resume`. That difference lives in the Assessment,
  never in Evidence.

**Persistence.** The saved document's frontmatter carries **both**: the immutable `evidence:`
(facts) and the `assessment:` that produced this record (the documentation judgment, kept for
provenance). Future purpose Assessments are computed on demand and are **not** stored in v0.2.
This rides the existing vault via `vault-layout.md` — no new storage mechanism (§7 respected).

- **Why split Assessment from Evidence:** immutable observations and context-dependent
  evaluations have different lifecycles. Facts are computed once and frozen; judgments are
  purpose-relative and may be recomputed or added. Naming both makes the multi-purpose future
  (§12) a matter of *adding an Assessment*, not reshaping the substrate.
- **Alternatives considered:** *(a) one combined Evidence-with-judgment object* (prior revision)
  — rejected: bakes the documentation purpose into the shared contract, so resume/interview must
  fight it. *(b) no persisted Assessment, recompute always* — rejected for the creating judgment:
  storing it is near-free and preserves provenance ("why was this saved"). Non-creating purposes
  are still recomputed on demand.
- **Why preferable:** one immutable fact-set, many cheap purpose-specific judgments — maximal
  reuse, clean lifecycles, no new layer.

## 5. Scoring model

Stage ③ reads the **Evidence object** (§4.1) and maps its contributing signals → **Importance
Score ⭐1–5**, reusing and sharpening the existing `importance-score.md` rubric. It does not
re-inspect the raw conversation for "why" — that judgment was made once, in Evidence.

**Decision:** *Rubric-anchored holistic judgment* — the model scores holistically but is
constrained by (a) explicit scoring dimensions (difficulty, reusability, interview value,
portfolio value, decision content — already in v0.1) and (b) calibration examples pinning what
each star level looks like.

- **Why:** we need scores that are *comparable across sessions* and *tunable* without code.
  Dimensions + anchors give the model a rubric to hold onto, reducing drift, while staying in
  prompt-space (no scoring engine to build).
- **Alternatives considered:**
  - *Numeric weighted sum of signals (e.g. difficulty×3 + decision×2…).* Rejected: false
    precision. In a prompt there is no reliable arithmetic substrate; weights can't be honestly
    tuned without real code and data, and the output *looks* exact while being arbitrary.
  - *Pure holistic "rate 1–5."* Rejected: drifts session to session; not evaluable.
- **Why preferable:** rubric-anchored judgment is the honest middle — structured enough to
  evaluate and calibrate (§11), loose enough to survive paraphrase and novelty. v0.2's core
  work is *expanding the calibration anchors* so ⭐⭐⭐ means the same thing every time.

**Output of stage ③:** a star score **plus a one-line rationale** naming the dominant signals
(e.g. "⭐⭐⭐⭐ — 대안 비교와 명확한 트레이드오프(ADR 성격), 면접 소재로 강함").

## 6. Confidence model

**Decision:** Confidence is a **separate axis** from importance, expressed at three levels —
**low / medium / high**.

Confidence is **derived from the Evidence object** (§4.1) and estimates two things:
1. **Boundary completeness** — read directly from `Evidence.resolution`: `confirmed` supports
   high confidence; `partial`/`none` caps it low. (Proposing to document an unsolved bug is the
   classic false positive.)
2. **Judgment reliability** — how clear-cut and corroborated are the contributing signals?
   One thin signal → low; several independent signals with concrete support → high.

- **Why a separate axis:** importance answers "is this valuable?"; confidence answers "are we
  sure, and is it done?" These are orthogonal. A ⭐⭐⭐⭐ event we caught mid-debug should **not**
  interrupt (high value, low confidence); a ⭐⭐ event we're certain about also shouldn't nag
  (low value, high confidence). Collapsing them loses exactly the information that prevents
  bad-timing interruptions — the #1 annoyance.
- **Alternatives considered:**
  - *Single combined score.* Rejected: conflates value with certainty; can't express "valuable
    but not yet."
  - *Continuous 0–1 confidence.* Rejected: false precision again, and harder to write testable
    policy against. Three coarse levels map cleanly to three actions (§7).
- **Why preferable:** three levels are enough to drive a clear interruption policy, are easy to
  label in eval fixtures, and keep the decision auditable.

## 7. False-positive prevention

Precision is the headline metric, so this gets explicit mechanisms rather than hoping the score
is good.

1. **Hard exclusion gate first (stage ①).** Trivial/non-dev work is rejected before any scoring.
   *Why:* cheapest possible precision; a typo never even reaches the scorer.
2. **Require ≥2 independent signals for an *unprompted* proposal.** A lone weak signal is not
   enough to interrupt. Explicit user request (§4A) bypasses this.
   *Why:* single-signal firing is the main over-trigger mode; corroboration raises precision at
   a small recall cost we accept (§3).
3. **Resolution required for proactive proposals.** No proposal without a §4F resolution signal
   — don't document unfinished work.
4. **Vault de-duplication.** If the topic is already recorded, stay silent (or, at most, note it
   once). *Why:* §5 one-concept-one-home extends to the user's vault; re-proposing known topics
   is noise.
5. **Per-session cooldown / cap.** After a decline, don't re-raise the same topic; cap
   unprompted proposals per session. *Why:* §3 attention; repeated asking erodes trust fast.
6. **Bias toward silence under low confidence.** When unsure, hold rather than ask.

- **Why this belongs in the engine, not the UI:** the constitution treats a false interruption
  as a defect, so prevention is a *design* property, not a cosmetic filter.
- **Alternatives considered:** *tune the score threshold higher and call it done.* Rejected: a
  single threshold can't distinguish "low value" from "high value but mistimed or duplicate."
  The mechanisms above target distinct failure modes.

## 8. User interruption policy

The policy is a small, explicit decision table over **(score, confidence, timing)**. Timing =
"are we at a natural boundary?" (problem resolved, task done, user changed topic).

| Situation | Action |
|-----------|--------|
| Explicit request / `/document` | **Propose** (bypass gates; still require approval to write) |
| Score ≥⭐⭐⭐ · confidence high · at a boundary | **Propose** once, in Korean |
| Score ≥⭐⭐⭐ · confidence low (mid-task/ambiguous) | **Hold** — re-evaluate at the next boundary |
| Score ⭐⭐ | **Stay silent** proactively (surfaces only if user asks) |
| Score ⭐ | **Stay silent** |
| Topic already in vault, or previously declined this session | **Stay silent** |
| Cap reached this session | **Stay silent** until user opts back in |

- **Why a table:** §6 wants unambiguous, testable instructions. A table is directly checkable
  against eval fixtures (§11) and leaves no room for two readings.
- **Why "hold and re-evaluate" exists:** it's the mechanism that turns a mistimed high-value
  catch into a *well-timed* one instead of a suppressed one — preserving recall without
  sacrificing timing.
- **Alternatives considered:** *always propose above a score threshold.* Rejected: ignores
  timing and confidence, producing exactly the mid-debug interruptions we most want to avoid.

**Proposal format** is unchanged from v0.1 (type · title · ⭐ · location · "기록할까요?"), so the
engine plugs into the existing user-facing surface with no new UI.

## 9. Classification handoff

When the decision is "propose," the engine emits a small, stable **detection result** and hands
off to the existing workflow. Conceptual contract (not a code type — a documented shape):

```
DetectionResult {
  evidence:   Evidence,     // §4.1 — immutable, purpose-independent facts
  assessment: Assessment    // §4.2 — purpose="documentation"; should_document, importance,
                            //        confidence, classification, explanation — all derived
  suggested_title: "…"
}
```

- **Clean separation:** `should_document`, score, confidence, type, and explanation all live in
  the Assessment (they are judgments); the Evidence carries only facts. The `explanation` is
  *rendered from* `evidence.contributing_signals`, so it can never disagree with the facts or the
  decision.
- Classification **reuses `references/doc-types.md`** tie-breakers, applied to the evidence — the
  engine does not reimplement classification. The handoff feeds SKILL.md step 4 (propose) → step 5
  (write, via `vault-layout.md`), and **both** the Evidence and this documentation Assessment are
  persisted into the saved document's frontmatter.
- **Why a defined result shape:** it is the seam that lets v0.4 hooks and future read-side
  features consume the same verdict without touching the engine (§8: integrate via a stable
  contract, not by coupling).
- **Alternatives considered:** *let detection flow straight into free-form proposal text.*
  Rejected: no reusable contract, so hooks/reporting would later have to re-derive the verdict.
- **Why preferable:** one small, explicit interface; everything downstream depends on the shape,
  not the internals.

## 10. Edge cases

| Case | Intended behavior | Rationale |
|------|------------------|-----------|
| Mid-debugging, not yet solved | Hold (low confidence, no resolution signal) | Don't document unfinished work |
| "Fixed" then regresses | Prefer proposing after *confirmed* resolution; if already proposed, allow an update rather than a duplicate | Accuracy over speed |
| One session, several documentable events | Evaluate each event separately; respect the per-session cap and propose the highest-value first | Avoid a barrage (§3) |
| Topic already in vault | Silent, or a single "이미 기록돼 있어요" note | §5 no duplication |
| Trivial work, user insists | Honor the explicit request (bypass), but still show the (low) score | User autonomy > our heuristic |
| Huge single event (whole feature) | Propose one coherent doc, or offer to split by sub-topic on approval | Keep records atomic and reusable |
| Pure exploration, no resolution | Silent unless the user asks | Nothing settled to preserve yet |
| Two types fit equally | Rank both in `type_candidates`; let doc-types.md tie-breakers decide; surface the pick | Reuse existing classification logic |
| User declined earlier | Cooldown; don't re-raise that topic | Trust (§3) |
| Non-dev chatter | Gate rejects at stage ① | Cheap precision |

## 11. Evaluation strategy

Because behavior is prompt-driven and non-deterministic (v0.1 limitation), we evaluate against
**labeled fixtures**, not unit tests.

- **Fixture set:** a folder of short, realistic conversation transcripts, each with a gold
  label: `should_document` (bool), expected `type`, and a rough expected `score` band.
  Deliberately include hard negatives (typos, mid-debug, duplicates) — the cases we most need to
  get right.
- **Metrics, in priority order:**
  1. **Precision** on `should_document` (headline — false-positive rate is the UX metric).
  2. **Recall** on clearly-valuable events (secondary; we accept some misses).
  3. **Classification accuracy** (type match).
  4. **Score agreement** (within ±1 star of the gold band).
- **Evaluate facts and judgments separately** (enabled by the Evidence/Assessment split):
  fixtures label the expected **Evidence** (which signals fired, resolution — objective, high
  inter-rater agreement) *and* the expected **Assessment** (should_document, type, score band —
  inherently more variable). Measuring Evidence-extraction accuracy apart from judgment quality
  localizes failures: a bad decision traceable to a *missed fact* is a different fix than one
  traceable to *mis-weighted judgment*. Conflating them (prior design) hid which was at fault.
- **Method:** run the engine's prompts against each fixture, compare to gold, review misses.
  Every real-world false positive becomes a new regression fixture.
- **Why fixtures over metrics dashboards or automated CI now:** §"don't over-engineer." A small,
  hand-runnable fixture set gives real signal at near-zero infrastructure cost; an automated
  harness can wrap the same fixtures later without redesign.
- **Alternatives considered:** *ship and eyeball it.* Rejected: precision claims need evidence,
  and fixtures are what make future prompt edits safe to land (no silent regressions).

This directly pays down the v0.1 debt item "no automated/eval tests" in the cheapest useful form.

## 12. Future extensibility

- **v0.4 hooks** call the identical engine (it's a pure decision function); only the trigger is
  new. No engine changes.
- **New signals / new types** are added as reference data, not code — the taxonomy and rubric
  grow; the pipeline shape doesn't.
- **Evidence is the metadata contract** (§8): immutable, purpose-independent facts persisted in
  each saved document's frontmatter — available to every future capability with no logging
  subsystem.
- **New purposes = new Assessments over the same Evidence.** Resume Score, Interview prep, and
  Portfolio (v0.5+) each add an Assessment (`purpose: "resume"` etc.) computed from the existing
  Evidence — no re-observation, no engine change. This is the concrete payoff of the fact/judgment
  split: the substrate is written once; evaluations multiply cheaply.
- **Reports (v1.0)** aggregate Evidence + Assessments across the vault's frontmatter — the read
  side never touches the engine.
- **Multi-language** later: signals are semantic, so the taxonomy ports; only output templates
  and cue-phrase examples are language-specific.

Nothing above requires a new layer — each is either more reference data or a new *caller* of the
existing contract.

## Recommended MVP (smallest thing that delivers real value)

Ship v0.2 as **prompt/reference upgrades + Evidence in frontmatter + a fixture set**, nothing
structural (no new runtime layer):

1. **Rewrite `detection-rules.md`** into the explicit **signal taxonomy (§4)** + the
   **hard-exclusion gate (§7.1)** + the **"≥2 signals or explicit request" rule (§7.2)** +
   **resolution-required (§7.3)**, and specify that stage ② emits the **Evidence object (§4.1)** —
   facts only, purpose-independent — with its three guardrails (qualitative, one-vocabulary, lean).
2. **Sharpen `importance-score.md`** with more **calibration anchors** so star levels are stable
   (§5), scoring explicitly from the Evidence object as part of the `documentation` **Assessment**.
3. **Add `references/interruption-policy.md`** (locked decision #1) holding the **3-level
   confidence (§6)** and the **decision table (§8)** — vault de-dup and a per-session
   "don't re-ask after a decline" cooldown included. SKILL.md only *points* to it, staying thin.
4. **Add `evidence:` and `assessment:` blocks to the template frontmatter** (§4.1/§4.2): immutable
   facts + the documentation judgment, and specify that the user-facing explanation is *rendered
   from* Evidence, not written independently (§9). This is the payoff of the amendments and is
   nearly free (frontmatter already exists). Only the `documentation` Assessment is produced now.
5. **Add top-level `tests/`** (locked decision #2) with `tests/fixtures/` — ~10–15 labeled
   transcripts (heavy on hard negatives), labeling **Evidence and Assessment separately** (§11),
   plus a one-page "how to run and score by hand."

**Explicitly deferred (to avoid overengineering):** numeric per-signal weighting, automatic hooks
(v0.4), a separate **decision log / analytics store** (Evidence-in-frontmatter is *not* that —
it's free persistence riding the saved doc), cross-session memory, and any automated eval
harness/CI. Continuous confidence and reporting are later.

**Why this is the right MVP:** it targets the one metric that makes the tool feel useful in real
sessions — *fewer, better-timed, better-scored proposals* — entirely by refining references the
skill already reads and adding test data. It changes no architecture, adds no layer, and leaves
every future feature (hooks, resume score, reports) a strict superset. It is faithful to
PROJECT_PRINCIPLES: thin orchestrator, one home per concept, precision over volume, and growth by
data rather than structure.
