# Hooks (v0.4.1)

자동 탐지 훅. **제안까지만 자동화하고, 저장은 여전히 사용자 승인이 필요합니다.**

## 왜 필요했나

판단 규칙(`references/`)은 다 있었는데 **제안이 거의 뜨지 않았습니다.**

스킬은 **모델이 스스로 켜는 것**이고, 그 방아쇠는 스킬 설명(description)입니다. 그런데 이
스킬의 발동 조건은 "개발 문제를 **다 풀고 난 뒤에** 쓰세요"예요. 이건 사용자 요청에 매핑되는
조건이 아닙니다. 요청이 없는 상태에서 모델이 *자발적으로* "이제 기록 가치를 판단해볼까?" 하고
떠올려야 하는데, **그걸 시켜주는 게 아무것도 없었습니다.** 엔진은 멀쩡한데 시동 장치가 없던
셈이고, 그래서 `/document`로 직접 부를 때만 동작했습니다.

## v0.4가 실패한 이유 (그리고 v0.4.1이 다른 이유)

v0.4는 **Stop 훅**에서 `decision: block` + `reason`으로 모델을 호출했습니다. 동작은 했는데
**쓸 수가 없었습니다.**

Claude Code는 차단된 Stop의 `reason`을 **사용자 화면에 렌더링합니다.** 그래서 "조용히 넘어가라,
훅이 돌았다는 말도 하지 마라"는 지시문 자체가 매 발동마다 화면에 에러처럼 찍혔습니다.
**침묵 경로가 애초에 존재하지 않았습니다.**

이건 튜닝으로 못 고칩니다. 구조적인 문제예요 — `Stop`은 Claude Code의 `hookSpecificOutput`
union 멤버가 아니라서 `additionalContext`(사용자를 거치지 않고 모델에만 닿는 채널)를 쓸 수
없습니다. **`PostToolUse`는 멤버입니다.** 그래서 이벤트를 바꿨습니다.

| | v0.4 (폐기) | v0.4.1 |
|---|---|---|
| 이벤트 | `Stop` | `PostToolUse` (`Edit\|Write\|NotebookEdit`) |
| 채널 | `decision: block` + `reason` | `hookSpecificOutput.additionalContext` |
| 사용자 노출 | **매번 노출됨** | 없음 |
| 쿨다운 기준 | `session_id` | **프로젝트 경로 + 시간** |

## 구성

| 파일 | 역할 |
|------|------|
| `detect-on-edit.sh` | 훅 본체. 값싼 게이트 → 통과 시 모델에게 조용히 판단을 요청 |
| `hooks.json` | 플러그인 설치용 훅 등록 (`${CLAUDE_PLUGIN_ROOT}` 사용) |

심링크 개발 설치는 `hooks.json`을 읽을 플러그인 매니페스트가 없어서, `scripts/install-dev.sh`가
`scripts/hook-wiring.py`로 `~/.claude/settings.json`에 직접 등록합니다. 이 스크립트는 v0.4
Stop 훅 잔재도 함께 제거합니다.

## 동작

```
파일 편집 (Edit / Write / NotebookEdit)
   │
   ▼
쿨다운 확인  ← 이미 걸려 있으면 stat 한 번으로 즉시 종료
   │
   ▼
값싼 게이트  ← "판단할 가치가 있나?" 만 판정
   │  편집 ≥3, 그리고 메시지 ≥12개
   ▼
additionalContext 주입 (사용자에게 안 보임)
   │
   ▼
모델이 작업 마무리 시점에 references/ 규칙으로 판단  ← "기록할 가치가 있나?" 는 여기서만
   │
   ▼
가치 있으면 → "기록할까요?" 제안 / 아니면 아무 말 없이 종료
```

### 역할 분리가 핵심

훅은 **기록 가치를 절대 판단하지 않습니다.** "작업이 실제로 일어났나?"만 봅니다. 가치 판단은
`references/`에 단 하나로 남아야 하고, 훅이 그걸 흉내내면 규칙이 두 군데로 갈라집니다.

그래서 주입되는 지시문에 못박아뒀습니다 — *"이 알림은 활동이 있었다는 것만 보고한다. 가치
판단이 아니며 `should_document`의 근거로 쓰면 안 된다. 침묵이 정상이고 가장 흔한 결과다."*

## 임계값

`detect-on-edit.sh` 상단에 상수로 모여 있습니다.

| 상수 | 값 | 의미 · 정한 이유 |
|------|-----|------------------|
| `MIN_EDITS` | 3 | 편집 횟수. v0.4의 1에서 올림 — `PostToolUse`는 편집마다 돌기 때문에 스쳐가는 편집 하나로 무장되면 안 됨 |
| `MIN_MESSAGES` | 12 | 너무 짧은 대화(한 줄 수정, 중단된 시작)를 걸러냄 |
| `COOLDOWN_HOURS` | 6 | 같은 프로젝트에서 다시 무장되기까지의 간격 |

전부 **실사용 데이터 없이 눈으로 정한 값**입니다. 진짜 판단은 엔진이 하므로 이 게이트는
"개발 작업이 아예 없는 세션"만 거르면 됩니다. `MIN_EDITS`와 `MIN_MESSAGES`는 **AND**입니다.

### 쿨다운을 세션이 아니라 프로젝트로 잡은 이유

v0.4는 `session_id`로 잡았고 그게 두 번째 결함이었습니다. **한 덩어리의 작업이 여러 세션에
걸쳐 있으면** — 백그라운드 잡과 인터랙티브 세션은 서로 다른 `session_id`입니다 — "세션당 1회"가
지켜져도 사용자 체감으로는 반복입니다. 실제로 마커에 UUID가 2개 찍혔습니다.

사용자가 체감하는 단위는 "지금 이 프로젝트"이므로, 마커를 **cwd 해시**로 잡고 시간으로
만료시킵니다. 같은 디렉터리에서 일하는 모든 세션이 하나를 공유합니다.

### 트랜스크립트를 읽는 방식

- **줄 수가 아니라 메시지 수를 셉니다.** JSONL에는 스냅샷·훅 기록·첨부 같은 관리용 줄이 섞여
  있습니다. 실제 세션에서 **564줄 vs 실제 메시지 309개** 였습니다.
- **슬럽하지 않고 스트리밍합니다.** `jq -s`는 149MB 트랜스크립트에서 **RSS 516MB**가 나왔습니다.
  줄 단위로 흘려보내면 **3.3MB**로 평평합니다. 편집마다 도는 훅이라 v0.4보다 훨씬 중요해졌습니다.
- 메시지의 `content`가 **문자열**인 경우도 있습니다(실제 세션에 13건). 툴 추출은 건너뛰고
  메시지 수에는 포함합니다.
- **쿨다운을 트랜스크립트 스캔보다 먼저 확인**합니다. 무장된 뒤의 편집은 `stat` 한 번(0.05초)만
  씁니다. 무장 전 전체 스캔은 2MB 기준 0.08초.

## 안전장치

지식 기록 도구가 사용자 세션을 망가뜨리면 안 됩니다. 그래서 **예상 못 한 모든 상황에서 조용히
exit 0** 합니다.

- **사용자 노출 채널을 절대 안 씀** — `decision` / `reason` / `systemMessage`를 내보내지
  않습니다. 테스트로 강제합니다(v0.4가 죽은 지점이라 설계 의도가 아니라 단언입니다).
- **stderr 억제** — 입력이 깨져도 에러가 사용자 터미널에 새지 않습니다.
- **jq 없으면 침묵** — 추측하지 않고 그냥 빠집니다.
- **마커 30일 정리** — 프로젝트당 파일 1개가 영구히 쌓이지 않도록.

## 끄기

```bash
export SECOND_BRAIN_HOOK_DISABLED=1   # 런타임에 비활성화
./scripts/install-dev.sh --no-hook    # 설치할 때 훅만 제외
./scripts/uninstall-dev.sh            # settings.json에서 제거
```

## 테스트

```bash
./tests/hook-gate.sh
```

게이트 판정 20개 케이스를 검증합니다. `tests/fixtures/`가 *엔진의 판단*을 검증한다면, 이쪽은
그 아래 계층인 *훅의 게이트*와 **출력 채널 계약**을 검증합니다.

## 상태 파일

프로젝트당 쿨다운 마커가 쌓입니다.

- 기본값: `~/.claude/second-brain-state/`
- 변경: 환경변수 `SECOND_BRAIN_STATE_DIR`

빈 파일이고 30일 지난 것은 자동 정리됩니다. 수동으로 지워도 안전합니다(해당 프로젝트에서
쿨다운이 즉시 풀릴 뿐입니다).
